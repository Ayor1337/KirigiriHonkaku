# 技术栈规范

> 本文档只记录当前已经确认的技术框架与基础设施选择，不展开到实现步骤与模块细节。
> 固定指令：Implementation Plan, Task List and Thought in Chinese

## 1. 定位

本文档用于回答：

- 这个项目的后端技术栈是什么
- 数据、AI、运行时文件各自采用什么基础设施
- 当前技术选择的边界和原则是什么

## 2. 总体结论

当前技术栈可以概括为：

**一个以 FastAPI 为后端入口、以 PostgreSQL 为结构化状态存储、以 SQLAlchemy 2.0 + Alembic 为数据层、以 LangChain + LangSmith 为 AI 层基础、并以独立 `data/` 目录承载运行时文本资料的单体后端方案。**

## 3. 后端框架

当前已经确认：

- 项目采用**单体后端**
- Web 框架使用 **FastAPI**

这样做的原因：

- 项目复杂度重点不在分布式
- FastAPI 足够承接接口层需求
- 与后续 AI 调用、异步接口和结构化返回兼容性较好

## 4. 数据层

当前已经确认：

- 数据库使用 **PostgreSQL**
- ORM 使用 **SQLAlchemy 2.0**
- 迁移工具使用 **Alembic**

这样做的原因：

- 项目存在较复杂、长期演化的结构化状态
- 需要稳定的关系建模与迁移能力
- 不希望后续再从轻量数据库迁移一次

## 5. 运行时文件

当前已经确认：

- 长文本资料不进数据库主存
- 运行时文本资料使用独立文件目录管理
- 文件根目录独立为 **`data/`**

这样做的原因：

- 结构化状态与长文本资料需要分治
- 代码目录、文档目录、运行时目录应分开
- 会话资料、对话全文、摘要、记忆文本更适合落盘保存

## 6. AI 技术栈

当前已经确认：

- AI 接入与编排使用 **LangChain**
- AI 观测与评测使用 **LangSmith**

当前不采用：

- LangGraph

### 6.1 LangChain 的定位

LangChain 当前用于承接：

- 模型接入
- Prompt 组织
- 工具调用
- 输出解析
- 对话摘要与记忆更新链路

### 6.2 LangSmith 的定位

LangSmith 当前用于承接：

- 调用追踪
- 运行时观察
- 调试分析
- 评测与质量回看

## 7. 同步与异步方向

当前已经确认：

- 接口层允许使用异步
- 引擎主流程按同步规则机思路设计

这意味着：

- FastAPI 路由可为 async
- 数据访问和 AI 调用可以使用 await
- 但 Game Engine 不应被设计成复杂的异步控制流

## 8. 项目结构方向

当前已经确认：

- 项目目录先按技术分层，而不是按领域分层

当前理解中的方向包括：

- `api/`
- `engine/`
- `ai/`
- `models/`
- `schemas/`
- `repositories/`
- `storage/`
- `core/`

这里的原则是：

- 目录按技术分层
- 规则边界仍按领域控制

## 9. 当前结论

当前技术栈可以先按下面这句话理解：

**这是一个以 FastAPI + PostgreSQL + SQLAlchemy 2.0 + Alembic 为规则与数据基础，以 LangChain + LangSmith 为 AI 基础，并通过 `data/` 目录承接运行时文本资料的单体后端技术方案。**
