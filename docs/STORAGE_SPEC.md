# 存储边界规范草案

> 本文档用于从 [GAME_SPEC.md](C:/Users/ayor/PycharmProjects/KirigiriHonkaku/docs/GAME_SPEC.md) 中拆出存储边界相关设计。
> 当前只记录数据库与文件系统的大框架分工，不展开实现级细节。
> 固定指令：Implementation Plan, Task List and Thought in Chinese

## 1. 文档定位

本文档当前只回答以下问题：

- 为什么这个项目需要数据库与文件系统混合存储
- 哪些内容更适合数据库
- 哪些内容更适合文件

## 2. 总体结论

当前存储系统可以概括为：

**这是一个数据库与文件系统混合存储的方案：数据库负责结构化世界状态，文件系统负责长文本叙事、记忆、对话与人类可读材料，两者通过索引与引用协同，而不是互相重复。**

## 3. 为什么需要混合存储

当前约定：

- 世界状态中有大量高结构化、频繁读写、适合查询判定的数据
- 同时也有大量长文本、叙事性强、需要人类可读和 AI 消费的内容

这意味着：

- 全部放数据库，会让长文本和调试体验变差
- 全部放文件，会让查询、推进和状态判定变得笨重

## 4. 数据库的职责

当前确认数据库必须覆盖：

- 会话元数据
- 玩家稳定层
- 玩家动态层
- 玩家认知层中的结构化部分
- 玩家工具层中的结构化部分
- NPC 当前状态与位置
- NPC 日程
- 地图图结构
- 线索实体
- 事件触发状态
- 暴露度与其他关键数值状态
- 对话索引

一句话理解：

- 数据库负责“系统要计算、查询、推进的东西”

## 5. 文件系统的职责

当前确认文件系统必须覆盖：

- `STORY.md` 这类开场叙事文本
- NPC 的 `PROFILE.md`
- NPC 的 `MEMORY.md`
- 对话摘要正文
- 对话完整记录正文
- 玩家自由笔记正文
- 侦探板自由文本内容
- 案件相关长文书
- 人类可读的会话导出 / 案卷材料

一句话理解：

- 文件系统负责“系统要保留、阅读、展示、回放的长文本内容”

## 6. 职责分离

当前已经明确以下内容应采用“职责分离、双向关联”的方式，不互相重复作为事实源：

- `NPC 状态` 与 `MEMORY.md`
- `对话索引` 与 `对话正文`
- `线索实体` 与 `长文书正文`

这意味着：

- 数据库不负责替代长文本正文
- 文件也不负责替代结构化状态主存
- 两者通过引用关系协作，而不是各写一套事实

## 7. 局目录的意义

当前总目录结构中，每局会保留独立目录，用于承载：

- 开场故事
- 世界历史
- 真相文档
- NPC 文本资料
- 对话文本
- 索引与导出材料

这意味着：

- 一局数据天然可隔离
- 便于调试、复盘、导出和人类查看

## 8. 当前结论

存储系统当前已经可以先按下面这句话理解：

这是一个“数据库管状态、文件管文本、局目录管归档、索引管连接”的混合存储方案，它的目标不是追求单一存储纯度，而是服务案件推进、AI 上下文、前端展示和调试复盘。

后续如果继续细化，优先讨论：

1. 哪些对象最适合以数据库为事实源
2. 哪些文本最适合做成标准化文件模板
3. 会话目录、索引和导出材料之间的关系
