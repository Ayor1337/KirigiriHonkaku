# Game Engine 规范

> 本文档只记录当前已经确认的 Game Engine 边界与职责，不展开到类设计、模块命名和实现细节。
> 固定指令：Implementation Plan, Task List and Thought in Chinese

## 1. 定位

Game Engine 是整个后端中的规则核心。

它的职责不是生成文本，也不是对外提供 HTTP 接口，而是：

- 接收动作
- 结算规则
- 推进世界状态
- 输出结构化结果

一句话：

- **API 收命令**
- **Engine 算结果**

## 2. 必须独立存在

当前已经确认：

- 必须存在单独的 Game Engine
- 游戏规则不应散落在 API、AI、存储层里

这样做的原因：

- 避免世界状态推进逻辑到处重复
- 避免 AI 文本结果反过来污染规则
- 避免 API 层承担不该承担的游戏逻辑

## 3. 核心职责

Game Engine 当前负责的内容包括：

- 时间推进
- 地图可达性判断
- NPC 位置与日程推进
- 线索存在与发现判定
- 暴露度主值变化
- 正式指认与结局判定

这些都属于硬规则和硬状态，必须由引擎控制。

## 4. 输入与输出

### 4.1 输入

Game Engine 接收的不是零散 HTTP 参数，而是统一动作。

当前方向是：

- 前端请求先进入 API
- API 再把请求翻译成统一的 `Action`
- Engine 统一处理 `Action`

### 4.2 输出

当前已经确认，Engine 每次执行动作后，至少应输出以下几类结果：

- 新的结构化世界状态
- 当前场景快照
- 需要 AI 生成的文本任务
- 本次动作的结构化结果摘要

这意味着：

- Engine 不直接负责最终文本
- Engine 负责把“世界发生了什么”算清楚

## 5. 与 AI 的边界

当前已经确认：

- Engine 负责规则与状态
- AI 负责表达与文本

具体到 Engine 这一层，就是：

- Engine 决定 NPC 当前知道什么范围
- Engine 决定哪些硬状态发生变化
- Engine 不负责写 NPC 的具体台词
- Engine 不负责写 `MEMORY.md` 正文

## 6. 状态边界

### 6.1 由 Engine 独占修改的硬状态

包括：

- 时间
- 玩家位置
- NPC 位置
- 地图可达性
- 线索存在 / 发现
- 暴露度主值
- 指认结果
- 结局状态
- NPC 日程主流程

### 6.2 不由 Engine 单独承担的软状态

以下内容可以在 Engine 结算之后，由 AI 运行时参与调整：

- NPC 对玩家态度
- NPC 警觉程度
- NPC 当前情绪标签
- 对话语气标签
- 对话标签
- `MEMORY.md` 文本
- 对话摘要

## 7. 执行顺序

当前已经确认，动作执行顺序必须是：

1. 玩家发起动作
2. API 转换为 `Action`
3. Engine 先结算硬状态
4. AI 再基于新状态生成文本
5. AI 在受限阶段调整软状态
6. 系统统一保存结构化状态与文本结果

当前不采用：

- AI 先生成，再让状态追文本

## 8. 内部结构方向

当前已经确认：

- Engine 不应是一个巨型类
- 应采用“核心引擎 + 多个规则模块”的结构

第一批规则模块至少包括：

- 时间模块
- 地图模块
- NPC 调度模块
- 线索模块
- 暴露度模块
- 指认 / 结局模块

## 9. 当前结论

当前 Game Engine 可以概括为：

**一个专门负责动作结算和世界状态推进的规则核心，它掌控所有硬状态，不直接生成文本，并通过统一动作输入与结构化结果输出衔接 API、AI 和存储。**
