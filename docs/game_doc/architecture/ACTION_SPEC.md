# 动作流转规范

> 本文档只记录当前已经确认的动作流转边界，不展开到接口路径、请求字段和类结构。
> 固定指令：Implementation Plan, Task List and Thought in Chinese

## 1. 定位

动作流转规范用于回答一个问题：

- 玩家的一次操作，是如何在后端中被接收、结算、表达和保存的

它的作用是把 API、Game Engine、AI Runtime 和存储之间的顺序固定下来。

## 2. 总体结论

当前动作流转可以概括为：

**前端动作先被 API 接收并转换为统一 Action，再由 Engine 结算硬状态，再由 AI Runtime 生成文本并调整软状态，最后统一保存结构化结果和文本结果。**

## 3. Action 的定位

当前已经确认：

- API 不直接改世界
- API 只把前端请求翻译成统一的 `Action`
- Engine 统一处理 `Action`

这意味着：

- 前端动作进入后端后，应先变成统一动作对象
- 动作是规则系统的输入，而不是零散 HTTP 参数直接进入引擎

## 4. Action 的作用

Action 的意义不是“记录请求”，而是“作为世界推进的标准入口”。

因此它至少承担以下意义：

- 表示玩家做了什么
- 进入引擎统一结算
- 触发时间推进与规则判定
- 形成可追踪的动作结果

## 5. 当前动作主轴

当前最核心的动作主轴仍然围绕以下几类：

- 移动
- 对话
- 调查
- 指认

这意味着：

- Action 系统不需要一开始覆盖所有复杂行为
- 先围绕主循环动作建立统一入口即可

## 6. 动作执行顺序

当前已经确认，动作执行顺序必须固定为：

1. 玩家发起动作
2. API 接收请求
3. API 转换为统一 `Action`
4. Engine 先结算硬状态
5. AI Runtime 基于新状态生成文本
6. AI Runtime 在受限阶段调整软状态
7. 系统统一保存结构化状态与文本结果

当前不采用：

- AI 先说，再让系统追认状态

## 7. Engine 的动作输出

当前已经确认，Engine 每次处理完 Action 后，至少应输出以下几类结果：

- 新的结构化世界状态
- 当前场景快照
- 需要 AI 生成的文本任务
- 本次动作的结构化结果摘要

这意味着：

- Engine 输出的是“结果骨架”
- AI Runtime 输出的是“结果表达”

## 8. 当前结论

当前动作流转可以先按下面这句话理解：

**Action 是玩家操作进入世界规则系统的统一入口，而一次动作的完整后端生命周期是：API 收动作、Engine 结算硬状态、AI 生成表达、系统统一保存结果。**
