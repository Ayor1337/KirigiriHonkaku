# 实现框架规范

> 本文档只记录当前已经确认的实现框架结论，不展开到接口、类设计和任务拆分。
> 固定指令：Implementation Plan, Task List and Thought in Chinese

## 1. 总体形态

- 项目采用**单体后端**
- 当前复杂度重点不在分布式，而在：
  - 状态一致性
  - 叙事组织
  - AI 与规则边界
- 因此当前不做多服务拆分

## 2. 核心边界

### 2.1 代码与 AI 的边界

- **代码负责规则与状态**
- **AI 负责表达与文本**

代码负责：

- 时间推进
- 地图可达性
- NPC 位置与日程推进
- 线索存在与发现
- 暴露度主值
- 正式指认与结局判定

AI 负责：

- NPC 说话内容
- NPC 说话风格
- 隐瞒、回避、态度表现
- 对话摘要
- `MEMORY.md` 的文本更新

### 2.2 NPC 表达边界

- 代码先决定 NPC 当前“知道什么范围”
- AI 再决定 NPC 在这个范围内“怎么说、说多少、怎么藏”

一句话：

- **代码管边界**
- **AI 管表演**

## 3. 执行顺序

玩家动作后的执行顺序为：

1. 先由代码结算硬状态
2. 再由 AI 基于新状态生成文本
3. AI 只在受限阶段调整软状态

当前不采用“AI 先生成，状态再追文本”的顺序。

## 4. 状态分层

### 4.1 硬状态

硬状态只允许引擎修改，包括：

- 时间
- 玩家位置
- NPC 位置
- 地图可达性
- 线索存在 / 发现
- 暴露度主值
- 指认结果
- 结局状态
- NPC 日程主流程

### 4.2 软状态

软状态可以考虑通过受限 AI 工具调整，包括：

- NPC 对玩家的态度
- NPC 警觉程度
- NPC 当前情绪标签
- 对话语气标签
- `MEMORY.md` 文本
- 对话摘要
- 对话标签

## 5. 后端主结构

后端主结构当前约定为：

- **API 收命令**
- **引擎算结果**
- **存储落状态**
- **AI 负责说出来**

## 6. Game Engine 定位

- 必须存在**单独的 Game Engine**
- 规则不应散落在 API、AI、存储层里
- Game Engine 采用：
  - 一个核心引擎负责流程
  - 多个规则模块负责子系统

当前已认可的第一批规则模块包括：

- 时间模块
- 地图模块
- NPC 调度模块
- 线索模块
- 暴露度模块
- 指认 / 结局模块

## 7. AI Runtime 定位

- AI 不是松散外挂能力
- AI 是单体后端中的**独立运行时子系统**
- 它在概念上与引擎、存储并列
- 但当前不拆成单独服务

## 8. 当前结论

当前实现框架可以概括为：

**一个单体后端，内部由 Game Engine 掌管规则和状态，由 AI Runtime 掌管角色表达和文本生成，两者严格分工、顺序执行。**
