# 状态边界规范

> 本文档只记录当前已经确认的状态分层与修改边界，不展开到数据模型、字段命名和存储实现。
> 固定指令：Implementation Plan, Task List and Thought in Chinese

## 1. 定位

状态边界规范用于回答两个问题：

- 哪些状态属于硬状态
- 哪些状态属于软状态

它的作用是防止引擎、AI 和存储层对同一类状态产生越权修改。

## 2. 总体结论

当前状态边界可以概括为：

**世界中的关键规则状态属于硬状态，只能由 Engine 结算；角色表达与互动氛围相关的辅助状态属于软状态，可以在受限边界内由 AI Runtime 参与调整。**

## 3. 硬状态

硬状态的特点是：

- 会影响世界规则推进
- 会影响案件客观结构
- 会影响后续所有关键判定

当前已经确认属于硬状态的内容包括：

- 时间
- 玩家位置
- NPC 位置
- 地图可达性
- 线索存在 / 发现
- 暴露度主值
- 指认结果
- 结局状态
- NPC 日程主流程

当前约定：

- 硬状态只允许由 Engine 修改
- AI Runtime 不直接决定这些状态

## 4. 软状态

软状态的特点是：

- 更接近角色表现与互动氛围
- 不直接决定世界硬规则
- 适合在动作结算之后做局部微调

当前可视为软状态的内容包括：

- NPC 对玩家的态度
- NPC 警觉程度
- NPC 当前情绪标签
- 对话语气标签
- 对话标签
- `MEMORY.md` 文本
- 对话摘要

当前约定：

- 软状态可以考虑由 AI Runtime 通过受限工具参与调整
- 但必须保持有限、可审计、可约束

## 5. 为什么要分层

当前状态分层的目的不是做学术区分，而是解决实际问题：

- 防止 AI 越权改世界
- 防止规则代码和文本代码互相污染
- 防止结算顺序混乱
- 保证案件世界仍有稳定事实源

## 6. 修改顺序

当前已经确认：

- 先结算硬状态
- 再处理软状态

这意味着：

- 世界规则先稳定下来
- AI 再在已确定的世界上做表达和微调

## 7. 当前结论

当前状态边界可以先按下面这句话理解：

**硬状态决定世界怎么运转，软状态决定角色怎么表现；硬状态归 Engine，软状态才允许 AI Runtime 在受限边界内参与。**
